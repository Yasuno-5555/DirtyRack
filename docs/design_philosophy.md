# DirtyData 核心的設計哲学 (Design Philosophy)

DirtyData は「音楽を作るためのソフトウェア」ではありません。「本質的にカオスな創作行為に対して説明責任（Accountability）を与えるためのランタイム環境」です。

このドキュメントでは、システムを支える核心的な思想について解説します。

## 1. State だけを持つな。History を持て。

従来の DAW（Digital Audio Workstation）は、セッションファイル（.als, .logicx 等）に「現在の状態（State）」のみを保存します。そのため、「なぜこのパラメーターがこうなっているのか？」という因果関係が時間と共に失われ、巨大なブラックボックスと化します。

DirtyData では、**すべての変更はパッチ（PatchSet）として表現され、有向非巡回グラフ（DAG）によって順序立てて適用**されます。
結果として、「今のこの音は、どのパッチがどのような意図で重なって出来上がったのか？」という **Provenance（来歴）** が常に証明可能です。

## 2. 操作ではなく「意味」を保存する

DirtyData は *Patch Manager* ではなく、*Meaning Manager* です。

ユーザーが「EQ の 2kHz を 3dB 上げた」という内部命令を記録するのではなく、「**ボーカルのプレゼンスを上げる（Vocal presence up）**」という **Intent（意図）** を保存します。
音楽は最適化問題ではなく、制約付き妥協問題です。Intent は以下のような Constraint（制約）としてグラフに紐付きます。

- `Must`: ボーカルのトランジェントを潰さない
- `Prefer`: アナログ的な暖かみを付加する
- `Avoid`: 高域の耳障りなシビランス
- `Never`: モノラル互換性の崩壊

システムは、パッチの適用時にこれらの Intent を評価し、「過去の自分が何を成し遂げようとしていたか」を後からレビューできるようにします。

## 3. Observe before Control (「疑い」の自動化)

オーディオシステムにおいて、GUI 上の数値と実際の出力音が一致している保証はどこにもありません（プラグインの隠しオーバーサンプリング、アナログモデリングのランダムノイズなど）。

DirtyData は、外部世界をコントロールする前に、まず **Observe（監視）** します。
ファイルサイズ、拡張子、そして BLAKE3 ハッシュを計算し、システムの内部状態（期待値）と外部世界（現実）の差異を常に評価します。

一致率（Confidence Score）は `Dirty State` として UI 上に明示されます。
- `100% Verified`: ハッシュが完全に一致する状態。
- `Suspicious`: 外部からの改ざんが検知された状態。

すべてを 100% Verified にすることは不可能（カオスの排除）ですが、DirtyData は **「どこが疑わしいか」を定量化し、人間がリスクを承知の上で作業を進めるための可視化** を行います。

## 4. 憲法: Explainable or Disposable

DirtyData の運用における絶対的なルールです：

> **すべてのノード、すべてのプラグイン、すべてのルーティングは、完全にその存在理由を説明可能（Explainable）であるか、さもなくばいつでも捨てられる（Disposable）状態でなければならない。**

`dirtydata doctor` コマンドは、この原則に違反している「存在理由が不明（Intent が紐付いていない）で影響度の低いノード」を Disposable Candidate（破棄候補）として警告します。

## 5. Security Model (信頼の境界)

DirtyData では、何が信頼できるか（Trust Boundary）を厳密に定義します。
システムは以下のものを **信用しません**：

1. **Plugin (VST/AU 等)**
2. **Observer (監視ツール自体)**
3. **GUI / ユーザー**

ユーザーが直接編集した DSL、外部スクリプト、サードパーティの拡張機能、または AI が自動生成した PatchSet は、すべて等しく `TrustLevel` を持ちます。
AI が勝手に「マスターバスにリミッターを 20 個挿す」といった破壊的行為（人類でもやりがちです）を行った場合、それは `ReviewRequired`（要人間レビュー）や `Quarantined`（隔離）として扱われ、不用意に本番環境を破壊することを防ぎます。

## 6. Performance Budget (性能予算)

「CPU は無料」という幻想を捨て、ドメインごとの性能予算（Performance Budget）を明確に定義します。

- **Sample Domain**: オーディオコールバック（Hard real-time）。アロケーション厳禁。ロック禁止。タイムアウト付きウォッチドッグによる監視。
- **Block Domain**: FFT やラウドネス解析（Soft real-time）。
- **Timeline Domain**: レンダリングや増分ビルドの管理。
- **Background Domain**: 機械学習やバッチ処理。この処理が Sample Domain をブロックすることは**絶対に許されません**。

## 7. 境界の防衛とフォールバック (Sandbox)

「音が出るブラックボックス（VST 等）」を制御するため、DirtyData はプラグインを別プロセスの Sandbox 内で実行します（IPC 境界）。

プラグインがメモリアクセス違反（Segfault）でクラッシュしたとしても、子プロセスが死ぬだけで **DirtyData コアは無傷** です。
プラグインが非正規化数（NaN Storm）を吐き出した場合、システムは境界でそれを検知し、出力を瞬時にミュートし、直前に保存されていた安全な `Frozen Asset`（レンダリング済み WAV など）にフォールバックします。

これが、DirtyData の目指す **真の堅牢性** です。

## 8. Visual as Projection (GUI は IR の「投影」である)

DirtyData において、GUI はシステムの状態そのものではなく、Core に存在する真実（IR）の「一時的な投影」に過ぎません。

- **The Forge UI**: 既存のノードエディタライブラリを採用せず、独自のレンダラーを構築した理由は、UI 自体に「DirtyState（不確実性）」を表現させるためです。
- **Shared Visualization**: 波形（Oscilloscope）や音量（Meters）は、システムの「内部動作の覗き見」であり、これによってユーザーはブラックボックス化しがちな DSP の挙動を、信頼できる来歴（History）と共に確認できます。
- **Direct Interaction**: GUI 上での操作（ドラッグ、結線）は、直ちに `UserAction` へと翻訳・コンパイルされ、パッチとして履歴に刻まれます。これにより、視覚的な直感とシステム上の厳密性が矛盾なく共存します。
